**Дата:** 22 июня 2025

## 📋 Краткое резюме

Через mitmweb перехват Windows приложения Glovis мы **точно определили** причину, почему сессия "умирает" после закрытия приложения.

## 🎯 Ключевые находки

### 1. **authToken привязан к активной сессии Windows приложения**

```
authToken: xMWEaaBPpJmiteLCzigMIw==
Base64 decoded: c4c58469a04fa499a2b5e2c2ce280c23 (16 байт)
```

- **Проблема:** Токен генерируется Windows приложением и действителен только пока приложение запущено
- **Решение:** authToken **НЕ МОЖЕТ** быть использован после закрытия приложения

### 2. **Критичные Windows-специфичные cookies**

```json
{
  "idChk": "1", // Флаг проверки идентичности
  "Chk": "7552" // Код проверки (может меняться)
}
```

- **Проблема:** Без этих cookies сервер не признает сессию валидной
- **Статус:** ✅ Успешно интегрированы в наш парсер

### 3. **Referer с authToken обязателен**

```
Referer: https://auction.autobell.co.kr/auction/exhibitList.do?authToken=xMWEaaBPpJmiteLCzigMIw%3D%3D&...
```

- **Проблема:** Сервер проверяет наличие authToken в Referer header
- **Статус:** ✅ Реализовано в новом сервисе

## 🔐 Структура authToken (16 байт)

```
Hex: c4c58469a04fa499a2b5e2c2ce280c23
     [--время--] [--хеш--] [--проверка--]
      4 байта     8 байт     4 байта
```

**Предположительная структура:**

1. **Timestamp** (4 байта): Время создания токена
2. **Hash/ID** (8 байт): Идентификатор сессии/пользователя
3. **Checksum** (4 байта): Контрольная сумма

## 🚨 Корень проблемы

**Windows приложение Glovis создает "закрытую" экосистему:**

1. **Приложение запускается** → Генерирует authToken + устанавливает cookies
2. **Браузер использует токен** → Успешная авторизация на сайте
3. **Приложение закрывается** → Токен инвалидируется на сервере
4. **Браузер теряет доступ** → "Требуется авторизация"
